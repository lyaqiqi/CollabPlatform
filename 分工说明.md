## 通用约定（四人都要遵守）

开工前先读一遍框架交付的三份文档：`docs/api-spec.md`、`docs/git-workflow.md`、`docs/getting-started.md`。下面是最关键的几条复述：

**Git 流程**
- 各自在自己的分支开发：`feature/board`(B) / `feature/doc`(C) / `feature/user`(D) / `feature/collab`(E)
- 不要直接 push main，开发完发 PR，至少一人 review 后合并
- commit 用前缀：`feat:` / `fix:` / `docs:`

**后端约定**
- 所有接口必须用框架的统一响应封装 `success()` / `fail()`，不要自己拼 JSON
- 需要登录的接口必须挂 `authMiddleware`，用户信息从 `req.user.userId` 取
- 业务报错 throw 框架的 `AppError`，由全局错误中间件处理，不要自己 try-catch 返回
- 接口路径遵循命名约定：小写、复数、连字符（如 `/api/items`、`/api/items/:id/versions`）
- 错误码用框架定义好的那套（40001 参数错 / 40101 未登录 / 40301 无权限 / 40401 不存在 / 40901 冲突 / 50000 服务器错）
- 各自的代码放进 `routes/` `controllers/` `services/` 下的**自己的文件**，不要改公共中间件和配置

**前端约定**
- 调后端用框架封装的 `api/request.js` 实例，它会自动带 token、统一处理错误，不要自己 new axios
- 实时通信用框架封装的 `socket/useSocket.js`，不要自己碰 socket.io-client 底层
- UI 统一用 antd，复用框架做好的 `Navbar`/`Loading`/`Toast`/`ConfirmDialog`，成员A多提供了一个DESIGN.md也可参考，总之要保证白板、富文本、登录认证、设置等前端界面的风格是统一的。
- 登录态从 `store/authStore.js` 读

**实时协作的统一思路（B、C、E 都涉及）**
- CRDT 统一用 **Y.js**。每个协作项目对应一个 Y.Doc。
- 客户端通过 `useSocket` 的 `joinRoom(itemId)` 进入房间，操作产生的 Y.js 更新通过 `emit` 发出，监听 `on` 接收别人的更新。
- 服务端用框架的 `broadcastToRoom(itemId, event, data, exceptSelf)` 把更新广播给房间内其他人。
- 三人在 `server/src/socket/handlers.js` 里**各自注册自己的事件**（白板事件 B、文档事件 C、评论事件 E），按框架预留的 TODO 模式扩展，互不冲突。

# 成员 B — 白板功能

负责整个共享白板模块，前后端一条龙。对应需求文档的 CollabBoard 子系统（UC-03、UC-04、UC-08 白板部分）。

## B-1 白板画布（单人版先跑通）

先不接实时同步，做出单人可用的画布。

- 在 `BoardPage.jsx` 集成 **Fabric.js**，初始化一个 canvas 画布，铺满编辑区。
- 工具栏（用 antd 按钮组）支持选择以下工具：
  - **矩形**、**圆形**：点击工具后在画布拖拽生成
  - **箭头**：拖拽生成带箭头的线
  - **手绘**：自由画笔模式
  - **文字**：点击画布插入可编辑文本框
- 元素操作：选中后可**拖拽移动**、**缩放**、**删除**（Delete 键 / 删除按钮）。
- **撤销/重做**：维护一个本地操作栈，支持 Ctrl+Z / Ctrl+Y（对应需求里至少 50 步历史）。
- 工具栏加颜色选择器、线宽选择（基础够用即可）。

**验收**：单人能在画布上画各种图形、移动删除、撤销重做，体验流畅（本地渲染延迟低）。

## B-2 白板实时同步（接入 Y.js + WebSocket）

把单人画布升级成多人实时同步。这是 B 最难的部分，留足时间。

- 引入 Y.js，为当前白板创建一个 `Y.Doc`，用一个 `Y.Map` 或 `Y.Array` 存所有画布元素（每个元素是一条记录，含 id、类型、坐标、尺寸、样式等）。
- 监听 Fabric 画布的增/删/改事件 → 转换成对 Y.js 数据的修改。
- 监听 Y.js 的 `update` 事件 → 通过 `useSocket.emit('board:update', { itemId, update })` 发给服务端。
- 监听 `useSocket.on('board:update', ...)` → 把别人的 Y.js 更新 apply 到本地 Y.Doc → 反映到 Fabric 画布。
- 进入页面时 `joinRoom(itemId)`，离开时 `leaveRoom`。
- **多人光标显示**：用 Y.js 的 awareness 机制（或自己发光标位置事件），把每个在线用户的光标位置 + 用户名 + 颜色实时画在画布上。

**服务端**：在 `socket/handlers.js` 注册 `board:update` 事件，收到后用 `broadcastToRoom(itemId, 'board:update', data, true)` 广播给房间内其他人（排除发送者）。

**验收**：两个浏览器窗口登录不同账号进入同一白板，一方画图，另一方 200ms 内看到；能看到对方的光标在动。网络抖动时 Y.js 自动合并不丢数据。

## B-3 白板内容持久化

- 设计一个保存机制：把当前 Y.Doc 序列化后存到后端，写入该项目 `CollaborativeItem.content_data`。
- 提供保存接口（如 `PUT /api/items/:id/content`，需 editor 以上权限——权限校验复用 D 做的中间件）。
- 进入白板时从 `content_data` 加载已有内容，恢复画布。
- 触发时机：可以防抖保存（如停止操作 2 秒后存）+ 离开页面前存。（注意与 E 的版本快照区分：这里是"当前最新状态"，E 那边是"历史快照"。）

**验收**：画完刷新页面，内容还在；新成员加入能看到已有内容。

## B-4 白板导出

- 工具栏加"导出 PNG"按钮。
- 调用 Fabric 画布的 `toDataURL()` 拿到图片数据，触发浏览器下载（生成 `<a download>` 点击）。
- SVG 导出为加分项，时间够再做。

**验收**：点导出能下载到当前白板的 PNG 图片。

## B 可交给 Claude Code 的部分
- Fabric.js 画布初始化、工具栏、各种图形绘制逻辑
- 撤销/重做栈实现
- Y.js 与 Fabric 的双向绑定胶水代码（这块较绕，让 AI 写初版你再调）
- PNG 导出
- 你需要自己把控的：实时同步的联调（多窗口测试）、与服务端事件名的对接

# 成员 C — 文档功能

负责整个共享文档模块。对应需求文档的 CollabDoc 子系统（UC-05、UC-08 文档部分、UC-11 创建文档）。**这是四个切片里相对省力的，因为 TipTap 的协作几乎开箱即用。**

## C-1 富文本编辑器（单人版先跑通）

- 在 `DocPage.jsx` 集成 **TipTap** 编辑器（基于 ProseMirror）。
- 支持以下格式（用 TipTap 官方 StarterKit 大部分都自带）：
  - 加粗、斜体、下划线、删除线
  - 标题 H1/H2/H3
  - 有序列表、无序列表
  - 代码块、行内代码
  - 引用块
- 做一个格式工具栏（antd 按钮），点击切换格式。
- 支持常用快捷键：Ctrl+B 加粗、Ctrl+I 斜体等（TipTap 自带）。

**验收**：单人能正常富文本编辑，各种格式可用。

## C-2 文档实时协作（TipTap + Y.js）

TipTap 官方有 Y.js 协作扩展，接入很顺：

- 安装 `@tiptap/extension-collaboration` 和 `@tiptap/extension-collaboration-cursor`。
- 为文档创建 `Y.Doc`，把 TipTap 的 Collaboration 扩展绑到这个 Y.Doc。
- **关键**：把 Y.js 的同步通道接到 A 封装的 `useSocket` 上。即监听 Y.Doc 的 update → `emit('doc:update', {itemId, update})`；监听 `on('doc:update')` → apply 到本地 Y.Doc。TipTap 会自动处理冲突合并，你几乎不用管 CRDT 细节。
- **彩色光标**：用 CollaborationCursor 扩展，传入当前用户名和随机颜色，自动显示每个协作者的光标和选区。
- 进入页面 `joinRoom(itemId)`，离开 `leaveRoom`。

**服务端**：在 `socket/handlers.js` 注册 `doc:update` 事件，`broadcastToRoom(itemId, 'doc:update', data, true)` 广播。

**验收**：两窗口同时编辑同一文档，输入实时同步（300ms 内），能看到对方彩色光标；同时编辑同一行不互相覆盖。

## C-3 创建文档（UC-11）

- 这个和 D 的项目管理有重叠：创建文档本质是创建一个 `type=Document` 的 CollaborativeItem。
- **建议**：创建项目的接口由 D 统一实现（见 D-2），C 这边负责创建后跳转到 `/doc/:id` 并初始化编辑器。和 D 约定好接口即可，别重复造。

## C-4 文档持久化

- 把 Y.Doc 序列化存到 `CollaborativeItem.content_data`（接口可与 B-3 共用同一个 `PUT /api/items/:id/content`，和 B、D 约定统一）。
- 进入文档时从 content_data 加载恢复。
- 需求里提到每 30 秒自动保存快照——那个是 E 做的版本快照；C 这里做的是"当前内容持久化"，防抖保存即可。

**验收**：编辑后刷新内容还在。

## C-5 文档导出 Markdown

- 加"导出 Markdown"按钮。
- 把 TipTap 内容转成 Markdown 文本（可用 `tiptap-markdown` 等扩展，或自己遍历 JSON 转），触发下载 `.md` 文件。
- PDF 导出为加分项，不强求。

**验收**：导出的 .md 文件用 Markdown 编辑器打开格式正确。

## C 可交给 Claude Code 的部分
- TipTap 编辑器集成、工具栏、格式按钮
- Collaboration + CollaborationCursor 扩展接入 useSocket 的胶水代码
- Markdown 导出
- 你需要自己把控的：和 D 约定创建/持久化接口、多窗口联调

# 成员 D — 用户与项目管理

负责账号体系和项目（白板+文档）的增删改查与权限。这是 B、C、E 都要依赖的基础，**要尽早做完接口**。对应需求文档 UC-01、UC-02、UC-09~UC-14。

## D-1 完善注册/登录（在 A 的鉴权骨架上）

A 已经做了基础的注册/登录示例，D 来完善成完整业务逻辑：

- **注册**：完善格式校验（邮箱合法性、密码长度≥8、用户名长度限制）、用户名和邮箱**查重**（已存在返回 40901）、密码 bcrypt 哈希。
- **登录**：完善错误提示（"邮箱或密码错误"），可选实现连续失败 5 次锁定 15 分钟（需求 UC-02 备选流，加分项，可用 Redis 记失败次数）。
- 前端完善 `LoginPage.jsx` 和 `RegisterPage.jsx`：表单、校验提示、提交后存 token（调 `authStore.setAuth`）、跳转首页。

**验收**：能正常注册、登录、登录后进首页；重复邮箱/弱密码有正确提示。

## D-2 项目管理接口（核心，B/C 依赖）

实现一套协作项目（白板/文档统一的 CollaborativeItem）的接口，全部挂 `authMiddleware`：

| 接口 | 方法路径 | 说明 | 权限 |
|------|----------|------|------|
| 创建项目 | `POST /api/items` | body 带 type(Whiteboard/Document)、title；创建后自动给创建者建一条 owner 权限 | 登录用户 |
| 我的项目列表 | `GET /api/items` | 返回当前用户拥有的 + 被分享参与的项目 | 登录用户 |
| 项目详情 | `GET /api/items/:id` | 返回项目信息（含 content_data） | 有权限者 |
| 重命名 | `PATCH /api/items/:id` | 改 title | 仅 owner |
| 删除 | `DELETE /api/items/:id` | 删项目（级联删版本/评论/权限） | 仅 owner |

- 删除前端要弹 `ConfirmDialog` 二次确认（UC-13）。
- 内容保存接口 `PUT /api/items/:id/content`（B、C 用）也由 D 统一实现，校验 editor 以上权限。

**验收**：能建白板/文档、列出、重命名、删除；非 owner 删改返回 40301。

## D-3 项目列表首页（UC-12）

- 完善 `HomePage.jsx`：
  - 调 `GET /api/items` 展示用户所有白板和文档（卡片或列表，区分类型图标）
  - "新建白板" / "新建文档"入口（调创建接口，成功后跳 `/board/:id` 或 `/doc/:id`）
  - 每个项目支持重命名、删除操作（调对应接口）
  - 加载中显示 Loading，失败显示重试提示（UC-12 备选流）

**验收**：首页能看到自己的项目，能新建、改名、删除，操作后列表实时刷新。

## D-4 权限系统（B/C/E 依赖）

实现基于角色的访问控制，对应需求的 owner/editor/viewer：

- 写一个**权限校验中间件 / 工具函数**（如 `checkPermission(itemId, userId, requiredRole)`），查 Permission 表判断用户对项目的角色是否达到要求。
- 把它做成可复用的，B（保存白板内容需 editor）、C（保存文档需 editor）、E（回滚需 owner）都会调用。
- WebSocket 侧也要能校验权限（用户加入房间、发更新前校验），和 A 的 socket 鉴权配合。

**验收**：viewer 不能编辑、不能删；editor 能编辑不能删；owner 全权限。

## D-5 分享链接 + 邮箱邀请（UC-09）

- **分享链接**：owner 在分享面板可开启链接分享并设公开权限（只读/可编辑），后端 `PATCH /api/items/:id` 改 is_public；生成带 token 的分享 URL，访客访问时校验。
- **邮箱邀请**：owner 输入对方邮箱，后端在 Permission 表插入对应角色记录（`POST /api/items/:id/permissions`），对方登录后能在列表看到。
- 实际发邮件是加分项，先做"填邮箱直接加为协作者"即可。
- 前端做一个分享面板（antd Modal）。

**验收**：owner 能邀请别人、设权限；被邀请者能看到并按权限协作。

# 成员 E — 协作增强功能

负责版本历史、评论批注、断线重连体验。这些功能横跨白板和文档，需要和 B、C、D 协调接口。对应需求文档 UC-06、UC-07。

## E-1 版本快照（UC-07 后端）

- **定时快照**：每 30 秒（或按需）把项目当前内容（content_data / Y.Doc 序列化结果）存一条记录到 Version 表（item_id、content_snapshot、created_at）。
  - 实现方式建议：可在服务端定时任务做，或在内容保存时触发。和 B、C 约定从哪里拿到内容快照。
  - 控制数量：每个项目只保留最近 50 条，超出删旧的（对应需求的历史步数）。
- 接口：
  - `GET /api/items/:id/versions`：返回版本时间轴列表（version_id + created_at）
  - `GET /api/versions/:vid`：返回某版本完整快照内容（用于预览）
  - `POST /api/items/:id/rollback`：回滚到指定版本（仅 owner，校验复用 D 的权限工具）

**验收**：编辑一段时间后能看到多条版本记录；预览能看到历史内容；回滚后内容变回去。

## E-2 版本历史前端（UC-07 前端）

- 在白板页和文档页加"版本历史"侧边栏（antd Drawer）：
  - 时间轴列表展示所有版本（时间戳）
  - 点击某版本，**只读预览**该时刻内容
  - owner 点"恢复到此版本"→ 调回滚接口 → 成功后通过 WebSocket 广播刷新通知给所有在线协作者，大家页面重新加载内容
  - 非 owner 点回滚提示"仅拥有者可执行"（UC-07 备选流）

**服务端**：回滚后用 `broadcastToRoom(itemId, 'version:rollback', ...)` 通知房间刷新。

**验收**：侧边栏能看历史、预览、回滚；回滚后其他在线成员页面自动刷新。

## E-3 评论/批注（UC-06）

主要针对文档（白板评论较复杂，可简化或作加分项）：

- **后端**：评论 CRUD 接口
  - `POST /api/items/:id/comments`：发评论，body 带 content、position（关联位置：文档锚点或白板坐标）
  - `GET /api/items/:id/comments`：列出该项目所有评论
  - `PATCH /api/comments/:cid`：标记已解决（is_resolved）/ 回复
  - 评论发出后通过 WebSocket 实时推送给在线成员（`broadcastToRoom(itemId, 'comment:new', ...)`）
- **前端**：
  - 选中文字 → 点"评论"按钮 → 弹输入框 → 发送
  - 右侧评论面板显示评论卡片（作者、时间、内容），高亮对应位置
  - 支持回复、owner 可"解决"折叠
  - 内容为空时发送按钮禁用（UC-06 备选流）

**验收**：选中文字能评论，右侧实时出现卡片并推送给其他人，能回复和标记解决。

## E-4 断线重连体验优化

A 的 useSocket 已经做了自动重连和状态管理，E 来完善**用户可见的体验**：

- 在界面显著位置（如顶部状态条）实时显示连接状态："已连接"/"已断开"/"正在重连"/"已恢复"（读 useSocket 暴露的状态）。
- 断线时本地操作暂存，重连成功后自动同步未发出的操作（Y.js 本身有离线缓存能力，配合做提示即可）。
- 重连后给用户一个"已恢复同步"的轻提示。

**验收**：手动断网，顶部提示"正在重连"，恢复后提示"已恢复"，断网期间的操作在恢复后同步出去。

## E 可交给 Claude Code 的部分
- 版本快照定时逻辑、版本接口、回滚
- 版本历史侧边栏 UI
- 评论 CRUD + 评论面板 UI + 实时推送
- 连接状态提示条
- 你需要自己把控的：和 B/C 约定怎么拿内容快照、和 D 约定权限校验、回滚广播的联调

# 模块间依赖与协调关系（重要）

四人不是完全独立，有几处必须约好接口再动手：

| 依赖 | 说明 | 谁约定 |
