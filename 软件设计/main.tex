\section{系统总体设计}

\subsection{项目背景与目标}

CollabPlatform 是一个面向课程实践的多人实时协作平台，目标是为小组协作提供两类核心工作空间：\textbf{共享白板}与\textbf{共享文档}。在典型的团队场景中，白板用于头脑风暴、流程梳理与原型草图；文档用于需求/方案/会议纪要的沉淀与多人联动编辑。系统采用前后端分离的 Web 架构，通过 HTTP 提供数据管理接口，通过 WebSocket 提供实时协作通道。

本项目的总体目标如下：
\begin{enumerate}[label=(\arabic*)]
  \item \textbf{协作一致性：}多人同时编辑时尽量避免冲突覆盖，保证内容可合并、可回滚。
  \item \textbf{可用性与易上手：}浏览器即可使用，界面统一（基于 antd），交互直观（工具栏、版本侧边栏、在线状态提示）。
  \item \textbf{工程化与可维护：}后端采用 ``routes $\rightarrow$ controllers $\rightarrow$ services'' 分层；前端采用模块化目录与统一封装（request/useSocket/authStore），便于多人并行开发与后续迭代。
  \item \textbf{安全与权限：}统一鉴权（JWT），并在业务层与 Socket 层对访问权限进行校验（Owner/Editor/Viewer）。
\end{enumerate}

\subsection{系统功能概述}

系统从用户视角提供以下主要功能：
\begin{enumerate}[label=(\arabic*)]
  \item \textbf{用户认证：}注册、登录、刷新 token、获取当前用户信息。
  \item \textbf{协作项目：}以 CollaborativeItem 为统一载体，支持 Whiteboard 与 Document 两种类型（同一套权限模型与版本/评论扩展能力）。
  \item \textbf{共享白板：}基于 Fabric.js 的画布编辑（选择、手绘、矩形/圆形/箭头/文字、拖拽缩放、删除、导出 PNG），支持多人协作与多人光标显示；并支持保存当前状态与版本快照、版本回滚。
  \item \textbf{共享文档：}基于 TipTap 的富文本编辑（标题、列表、代码块、表格、图片、链接等），基于 Yjs 的 CRDT 协作编辑与协作光标；支持评论/回复、版本快照与回滚、导出 Markdown/HTML 等扩展能力。
  \item \textbf{实时协作基础设施：}统一的 Socket 连接管理、房间管理（item:<itemId>）、广播与断线重连状态提示。
\end{enumerate}

系统采用统一响应格式（\texttt{\{code,data,message\}}）与错误码约定（40001/40101/40301/40401/40901/50000），前端通过 axios 拦截器统一解包与错误提示，减少业务层重复代码。

\subsection{系统总体架构设计}

\subsubsection{架构风格说明}

系统整体采用 \textbf{B/S（Browser/Server）} 架构与 \textbf{前后端分离} 模式，前端为单页应用（SPA），后端提供 RESTful 风格的 HTTP API 与 Socket.io 实时通道。架构中同时存在两条数据通路：
\begin{enumerate}[label=(\arabic*)]
  \item \textbf{HTTP（请求-响应）：}用于登录注册、项目数据增删改查、版本/评论等需要落库的操作，强调幂等与可追踪。
  \item \textbf{WebSocket（长连接）：}用于实时协作同步（文档/白板的增量更新、协作光标/在线状态等），强调低延迟与广播效率。
\end{enumerate}

后端内部采用分层架构以提升可维护性与可测试性：
\begin{center}
\begin{tabular}{c}
routes（声明路径）$\rightarrow$ controllers（处理 req/res）$\rightarrow$ services（业务逻辑）$\rightarrow$ Prisma（数据访问）$\rightarrow$ PostgreSQL
\end{tabular}
\end{center}

在实时协作层面，文档协作使用 \textbf{Yjs（CRDT）} 实现无冲突合并；白板协作采用 ``Fabric 事件 $\leftrightarrow$ Yjs 数据结构'' 的双向绑定，将画布元素抽象为可合并的数据更新，从而在多人并发操作时避免简单 ``整页快照覆盖'' 带来的冲突。

\subsubsection{系统整体结构图}

\begin{verbatim}
Browser (React + Vite)
  |
  |-- HTTP (axios) ------------------> Express /api/*
  |                                     |
  |                                     |-- routes -> controllers -> services
  |                                     |-- Prisma -> PostgreSQL
  |
  |-- WebSocket (socket.io-client) ---> Socket.io Server
                                        |
                                        |-- 连接鉴权（JWT）
                                        |-- 房间：item:<itemId>
                                        |-- 广播：broadcastToRoom（排除发送者）
                                        |-- 协作事件：doc:* / board:* / cursor
\end{verbatim}

其中，前端通过 \texttt{request.js} 统一封装 HTTP 请求（自动附带 accessToken、统一错误处理）；通过 \texttt{useSocket} 统一封装 WebSocket（单例连接、自动重连、连接状态枚举、join/leave/ping 等通用方法）。

\subsection{技术选型说明}

\subsubsection{开发语言、框架、数据库与中间件}

\textbf{前端：}
\begin{itemize}
  \item React 18 + Vite：构建单页应用与工程化开发。
  \item react-router-dom v6：路由与受保护页面跳转（ProtectedRoute）。
  \item antd：统一 UI 组件库与交互规范。
  \item Zustand：轻量状态管理（登录态持久化到 localStorage）。
  \item axios：HTTP 客户端与拦截器封装。
  \item socket.io-client：实时连接与自动重连。
  \item TipTap（ProseMirror）：文档富文本编辑器。
  \item Yjs + y-protocols + y-prosemirror：CRDT 协作内核与 Awareness（协作光标/在线用户）。
  \item Fabric.js：白板画布与图形对象模型。
\end{itemize}

\textbf{后端：}
\begin{itemize}
  \item Node.js + Express：HTTP API 服务与中间件体系。
  \item Prisma：ORM 与数据库模型管理。
  \item PostgreSQL：关系型存储（可部署于 Supabase）。
  \item Socket.io：实时通信与房间广播。
  \item jsonwebtoken + bcrypt：鉴权与密码哈希。
  \item Redis（可选）：缓存/限流/计数等能力，未配置时自动降级。
  \item morgan：请求日志。
\end{itemize}

\subsubsection{选型理由分析}

\begin{enumerate}[label=(\arabic*)]
  \item \textbf{协作一致性：}Yjs 作为成熟的 CRDT 方案，能够在弱网络与并发输入下实现无冲突合并；Awareness 机制可自然承载光标/在线信息同步。
  \item \textbf{开发效率：}React + Vite 具备快速热更新与良好生态；antd 提供覆盖全面的组件，减少重复造轮子。
  \item \textbf{可维护性：}Express 的中间件模式与 ``routes/controllers/services'' 分层清晰；Prisma 通过 schema 管理数据模型与迁移，降低手写 SQL 的出错概率。
  \item \textbf{实时与可靠：}Socket.io 兼容性好且内置重连机制，适合课程项目的多环境部署；房间广播可将协作范围限制在单项目内，降低不必要的网络开销。
  \item \textbf{可扩展：}CollaborativeItem 抽象统一了 ``白板/文档'' 两类资源，后续新增协作类型（如表格、脑图）只需扩展 type 与对应模块即可复用权限、版本、评论等基础能力。
\end{enumerate}

\section{详细设计}

\subsection{模块划分与模块功能设计}

\subsubsection{模块职责说明}

系统按照前后端分离的结构，将功能划分为若干具有清晰职责边界的模块，模块之间通过明确定义的接口交互。

\textbf{后端模块划分（\texttt{server/src/}）：}

\begin{enumerate}[label=(\arabic*)]
  \item \textbf{配置层（\texttt{config/}）：}负责运行环境初始化，是整个服务启动的前置依赖。
  \begin{itemize}
    \item \texttt{env.js}：加载 \texttt{.env}，校验 \texttt{DATABASE\_URL}、\texttt{JWT\_SECRET} 等关键变量；缺失时直接退出进程，防止以错误配置启动。
    \item \texttt{prisma.js}：导出 PrismaClient 单例，开发模式下挂到 \texttt{global}，避免热重载时重复建立数据库连接。
    \item \texttt{redis.js}：连接 Redis；若 \texttt{REDIS\_URL} 为空或连接失败，导出空操作 Proxy 进行容错降级，业务层无需感知 Redis 是否可用。
  \end{itemize}

  \item \textbf{工具层（\texttt{utils/}）：}提供全局通用能力，供各层调用。
  \begin{itemize}
    \item \texttt{AppError.js}：自定义错误类，携带 \texttt{httpStatus}、业务 \texttt{code}、\texttt{message} 三个字段；业务层 \texttt{throw} 后由错误中间件统一捕获并格式化返回。
    \item \texttt{response.js}：封装 \texttt{success(res, data)} 与 \texttt{fail(res, status, code, msg)} 两个函数，所有接口必须通过它们返回，保证响应格式一致。
    \item \texttt{jwt.js}：封装 \texttt{signAccessToken}、\texttt{signRefreshToken}、\texttt{verifyToken}；有效期通过环境变量 \texttt{JWT\_EXPIRES\_IN} / \texttt{JWT\_REFRESH\_EXPIRES\_IN} 配置；校验失败直接抛 \texttt{AppError(401)}。
  \end{itemize}

  \item \textbf{中间件层（\texttt{middlewares/}）：}处理横切关注点，以管道形式插入 Express 请求链路。
  \begin{itemize}
    \item \texttt{requestLogger.js}：基于 morgan，记录每个请求的方法、路径、状态码与耗时。
    \item \texttt{auth.js}：JWT 鉴权中间件，从 \texttt{Authorization: Bearer <token>} 头提取并校验 token，成功后将 \texttt{\{ userId \}} 挂载到 \texttt{req.user}，供控制器读取。
    \item \texttt{errorHandler.js}：全局错误处理，放在所有路由之后；捕获 \texttt{AppError} 按其字段格式化返回，未知错误统一返回 50000，开发环境输出堆栈信息。
  \end{itemize}

  \item \textbf{路由层（\texttt{routes/}）：}只负责声明 URL 路径与 HTTP 方法，将请求转发给对应 controller，本身不含任何业务逻辑。包含 \texttt{auth.routes.js}（认证）、\texttt{boards.routes.js}（白板）、\texttt{doc.routes.js}（文档）、\texttt{items.routes.js}（项目管理）、\texttt{health.routes.js}（健康检查），统一在 \texttt{routes/index.js} 挂载到 \texttt{/api} 前缀。

  \item \textbf{控制器层（\texttt{controllers/}）：}负责处理 HTTP 请求与响应，从 \texttt{req} 中提取参数、调用 service，再通过 \texttt{response} 工具返回结果；不包含业务逻辑判断。包含 \texttt{auth.controller.js}、\texttt{board.controller.js}、\texttt{doc.controller.js}、\texttt{item.controller.js}。

  \item \textbf{业务层（\texttt{services/}）：}核心业务逻辑所在层，只负责业务计算、数据库操作与权限校验，不感知 HTTP，可被多个 controller 复用，也方便编写单元测试。
  \begin{itemize}
    \item \texttt{auth.service.js}：注册（字段校验、唯一性检查、bcrypt 哈希）、登录（密码比对、状态检查、token 签发）、token 刷新。
    \item \texttt{board.service.js}：白板 CRUD，提供 \texttt{assertBoardReadable} / \texttt{assertBoardWritable} / \texttt{assertBoardOwner} 三个权限断言函数，以及版本快照管理（保留最近 50 条）。
    \item \texttt{doc.service.js}：文档 CRUD，提供 \texttt{assertDocumentAccess(itemId, userId, minRole)} 权限断言，以及评论（含回复树与锚点定位）、版本快照与回滚、成员邀请与权限调整的完整业务逻辑。
    \item \texttt{item.service.js}：协作项目的通用管理操作，支持按类型筛选列表与权限更新。
  \end{itemize}

  \item \textbf{Socket 层（\texttt{socket/}）：}实时协作通信层，与 HTTP 层并行工作。
  \begin{itemize}
    \item \texttt{index.js}：初始化 Socket.io 并与 HTTP Server 绑定；在 \texttt{io.use()} 中间件中对握手 token 进行 JWT 校验，校验通过后将 \texttt{userId} 挂载到 \texttt{socket.data}；导出 \texttt{joinRoom}、\texttt{leaveRoom}、\texttt{broadcastToRoom}（排除发送者）三个通用方法。
    \item \texttt{handlers.js}：为每条连接注册业务事件，包括房间加入/离开（查询 item 类型后分发到白板或文档权限校验）、文档协作事件（\texttt{doc:operation} / \texttt{doc:cursor} / \texttt{doc:title-changed} / \texttt{doc:sidebar-changed}）、白板协作事件（\texttt{board:sync} / \texttt{board:draw} / \texttt{board:cursor}）。每个写操作事件均在广播前调用对应 service 的权限断言函数。
  \end{itemize}
\end{enumerate}

\textbf{前端模块划分（\texttt{client/src/}）：}

\begin{enumerate}[label=(\arabic*)]
  \item \textbf{HTTP 封装（\texttt{api/}）：}基于 axios 创建实例，请求拦截器自动注入 \texttt{Authorization} 头；响应拦截器统一解包（\texttt{code === 0} 直接返回 \texttt{data}）、错误提示（Toast）与自动登出（\texttt{code === 40101}）。按业务领域进一步封装：\texttt{auth.api.js}（注册/登录/刷新）、\texttt{board.api.js}（白板 CRUD）、\texttt{doc.api.js}（文档 CRUD/评论/版本/成员）。
  \item \textbf{WebSocket 封装（\texttt{socket/useSocket.js}）：}单例模式封装 socket.io-client，保证多组件共享同一连接；暴露 \texttt{connect}、\texttt{disconnect}、\texttt{joinRoom}、\texttt{leaveRoom}、\texttt{emit}、\texttt{on}、\texttt{off}、\texttt{ping} 等方法；维护 \texttt{status} 枚举（\texttt{disconnected / connecting / connected / reconnecting}），供 \texttt{App.jsx} 渲染全局连接状态横幅。
  \item \textbf{协作层（\texttt{collab/}）：}文档 CRDT 协作内核，不依赖第三方协作服务器。\texttt{socketIoYjsProvider.js} 将 Yjs 增量 update 与 Awareness 编码为 base64 后经 \texttt{doc:operation} / \texttt{doc:cursor} 事件收发；\texttt{yjsUtils.js} 提供 Yjs 状态与 base64 的互转以及历史快照注水工具；\texttt{userColor.js} 根据 userId 哈希派生协作光标颜色。
  \item \textbf{状态管理（\texttt{store/authStore.js}）：}Zustand store，包含 \texttt{user}、\texttt{accessToken}、\texttt{refreshToken} 三个字段及 \texttt{setAuth}（同步写 localStorage）、\texttt{logout}（清空 state 与 localStorage）、\texttt{loadFromStorage}（页面刷新后恢复登录态，在 \texttt{App.jsx} 初始化时同步调用，无竞态问题）三个方法。
  \item \textbf{路由（\texttt{router/}）：}react-router-dom v6 配置五条路由，\texttt{ProtectedRoute} 检查 \texttt{accessToken}，未登录时重定向至 \texttt{/login}。
  \item \textbf{页面（\texttt{pages/}）：}各路由对应的顶层页面组件：\texttt{LoginPage}（登录）、\texttt{RegisterPage}（注册）、\texttt{HomePage}（协作项目列表）、\texttt{BoardPage}（白板编辑，基于 Fabric.js）、\texttt{DocPage}（文档编辑，集成 TipTap + Yjs）。
  \item \textbf{组件（\texttt{components/}）：}可复用 UI 组件。公共组件：\texttt{Navbar}（顶部导航栏）、\texttt{Loading}（全屏加载）、\texttt{Toast}（全局消息提示）、\texttt{ConfirmDialog}（危险操作确认弹窗）。文档专属：\texttt{DocEditor.jsx}（TipTap 富文本编辑器）、\texttt{DocCollabSidebar.jsx}（评论/版本/成员侧边栏）及编辑器浮层（气泡菜单、斜杠命令菜单、表格操作菜单）。
  \item \textbf{自定义 Hook（\texttt{hooks/}）：}\texttt{useDocCollaboration.js} 负责创建 \texttt{Y.Doc}、\texttt{SocketIoYjsProvider}、\texttt{Y.UndoManager}，加入协作房间，并从 Awareness 派生去重后的在线用户列表，是文档协作功能的核心入口。
\end{enumerate}

\subsection{核心业务流程设计}

\textbf{（1）用户注册与登录流程}

用户注册时，前端向 \texttt{POST /api/auth/register} 提交 \texttt{username}、\texttt{email}、\texttt{password}；服务端依次进行格式校验（邮箱正则、用户名长度不超 32 字符、密码不少于 8 位）、唯一性校验（查库判断邮箱或用户名是否已注册），通过后用 bcrypt（salt rounds = 10）对密码进行哈希存储，将用户记录写入 \texttt{User} 表，返回 201 与用户信息（不含密码哈希）。

用户登录时，前端向 \texttt{POST /api/auth/login} 提交邮箱与密码；服务端按邮箱查找用户，用 \texttt{bcrypt.compare} 验证密码哈希，检查账号是否被封禁，全部通过后签发 \texttt{accessToken} 与 \texttt{refreshToken}（有效期均通过环境变量配置）并返回。前端通过 \texttt{authStore.setAuth()} 将登录态持久化到 localStorage，后续每次 HTTP 请求由 axios 拦截器自动附带 \texttt{Authorization: Bearer <accessToken>}。

当 accessToken 过期时，axios 响应拦截器捕获错误码 40101，调用 \texttt{POST /api/auth/refresh} 用 refreshToken 换取新 accessToken；若 refreshToken 也已失效，则自动清理登录态并跳转至 \texttt{/login}。

\textbf{（2）文档实时协作流程}

\begin{enumerate}[label=\arabic*.]
  \item 用户打开文档页时，前端先通过 \texttt{GET /api/docs/:id} 拉取文档详情，获取 \texttt{content\_data.yjs\_state}（base64 编码的 Yjs 全量状态快照）。
  \item \texttt{useDocCollaboration} Hook 创建本地 \texttt{Y.Doc} 实例与 \texttt{SocketIoYjsProvider}，并通过 \texttt{useSocket.joinRoom(itemId)} 触发 Socket \texttt{join} 事件加入协作房间（服务端校验 viewer 权限）。
  \item 用 \texttt{applyPersistedYjsState(ydoc, yjs\_state)} 对文档进行注水（origin 标记为 \texttt{'persisted'}，不触发 Socket 广播），完成文档初始化渲染。
  \item 此后每次本地编辑触发 \texttt{Y.Doc} 的 \texttt{update} 事件，Provider 将增量 update 编码为 base64 后通过 \texttt{doc:operation} 事件发给服务端；服务端校验 editor 权限后，调用 \texttt{broadcastToRoom} 广播给房间内其他成员（排除发送者）。
  \item 远端成员收到 \texttt{doc:operation} 事件后，Provider 调用 \texttt{Y.applyUpdate} 将增量无冲突合并到本地 Y.Doc，TipTap 编辑器自动更新视图。
  \item 协作光标通过 \texttt{doc:cursor} 事件同步 Awareness（包含用户名、颜色、光标位置），所有在线成员的光标实时展示；在线用户列表由 Awareness 状态去重派生（同一用户多标签打开只计一次）。
  \item 文档内容在两种时机持久化：停止编辑 2s 后防抖自动保存，或页面卸载前兜底保存，均调用 \texttt{PUT /api/docs/:id} 将 \texttt{Y.encodeStateAsUpdate(ydoc)} 的 base64 写入数据库。实时编辑全程不走 HTTP，仅通过 Socket 增量同步。
\end{enumerate}

\textbf{（3）白板实时协作流程}

\begin{enumerate}[label=\arabic*.]
  \item 用户进入白板页，前端通过 \texttt{GET /api/boards/:id} 加载已保存的画布 JSON（\texttt{content\_data.canvas}），交由 Fabric.js 初始化渲染。
  \item 建立 Socket 连接后发送 \texttt{join} 事件加入房间，服务端查询 \texttt{CollaborativeItem} 类型后对白板执行 \texttt{assertBoardReadable} 权限校验。
  \item 用户在画布上操作（绘制、移动、缩放、删除元素）时，Fabric.js 事件触发，前端将当前完整画布序列化为 JSON，通过 \texttt{board:sync} 事件广播给房间内其他成员，其他成员收到后调用 \texttt{canvas.loadFromJSON} 重新加载画布。
  \item 用户光标位置通过 \texttt{board:cursor} 事件实时广播（节流处理），其他成员以带用户标识的小圆点展示光标。
  \item 停止操作 1.2s 后触发防抖自动保存，调用 \texttt{PUT /api/boards/:id} 将画布 JSON 落库，editor 及以上权限方可执行保存操作。
\end{enumerate}

\textbf{（4）版本快照与回滚流程}

\begin{enumerate}[label=\arabic*.]
  \item 用户点击「创建版本」，前端调用 \texttt{POST /api/docs/:id/versions}，将当前 Yjs 全量状态（base64）、文档标题与用户自定义标签写入 \texttt{Version} 表；服务端保留最近 50 条版本记录，自动删除更早的快照，并通过 Socket 广播 \texttt{doc:sidebar-changed} 通知其他成员刷新版本列表。
  \item 用户点击「回滚到此版本」，前端调用 \texttt{POST /api/docs/:id/versions/:versionId/restore}；服务端将该版本的 \texttt{yjs\_state} 覆写到文档 \texttt{content\_data}，然后主动向房间内\textbf{所有}成员（含操作者）广播 \texttt{doc:version-restored} 事件。
  \item 全员收到 \texttt{doc:version-restored} 后，前端执行页面 reload，重新拉取文档详情并注水，从而完成所有在线成员同步到指定历史版本。
\end{enumerate}

\subsection{安全性设计}

\begin{enumerate}[label=(\arabic*)]
  \item \textbf{认证与会话：}系统使用 JWT（accessToken + refreshToken）进行无状态认证；前端在请求拦截器中自动附带 \texttt{Authorization: Bearer <token>}，后端在 \texttt{authMiddleware} 中校验并将用户标识挂载到 \texttt{req.user}。
  \item \textbf{密码安全：}用户密码在服务端使用 bcrypt 进行哈希存储，不以明文形式落库。
  \item \textbf{权限控制：}采用 Owner/Editor/Viewer 三级权限模型。HTTP API 在 service 层做读写权限校验；Socket 层在 join 与协作事件处理中进行权限校验，避免绕过 HTTP 直接通过 WebSocket 读写数据。
  \item \textbf{跨域与最小暴露：}后端 CORS 仅放行配置的前端域名（CLIENT\_ORIGIN），并设置 \texttt{credentials}；统一错误码返回，避免泄露内部堆栈信息（生产环境不输出详细错误）。
\end{enumerate}

在生产部署时，建议进一步补充：HTTPS/WSS、访问频率限制（可基于 Redis 计数）、更严格的输入校验与审计日志等。

\subsection{性能优化设计}

\begin{enumerate}[label=(\arabic*)]
  \item \textbf{网络层：}实时协作使用房间广播，避免全局广播；对光标等高频事件采用节流；对持久化保存采用防抖，减少频繁落库。
  \item \textbf{协作层：}基于 Yjs 的增量 update 同步，仅传输差异数据，降低带宽占用并提升多人并发下的合并效率。
  \item \textbf{数据层：}Prisma 查询使用字段选择（select）与必要的索引/约束，降低无效字段读取；版本快照采用 ``只保留最近 N 条'' 的策略避免表无限增长。
  \item \textbf{前端渲染：}将连接状态、加载状态、侧边栏等 UI 状态与业务状态分离，减少不必要的重渲染；大型模块可按路由拆分实现按需加载（后续优化方向）。
\end{enumerate}

\subsection{可扩展性设计}

\begin{enumerate}[label=(\arabic*)]
  \item \textbf{模块化目录与职责边界：}前端按 api/components/pages/hooks/collab 分层，后端按 routes/controllers/services 分层；新增功能通常只需要在对应层新增文件并在路由入口注册。
  \item \textbf{统一协作模型：}CollaborativeItem 作为统一资源抽象，权限（Permission）、版本（Version）、评论（Comment）为通用扩展点，新协作类型可复用这些能力。
  \item \textbf{统一协议与常量：}错误码与 Socket 事件名集中定义，减少多人协作时的命名分歧与联调成本。
\end{enumerate}

\subsection{异常处理机制}

\begin{enumerate}[label=(\arabic*)]
  \item \textbf{后端：}业务层通过抛出 \texttt{AppError(httpStatus, code, message)} 表达可预期错误；全局错误中间件统一捕获并按标准格式返回。对未知错误返回 50000，开发环境可输出日志用于定位。
  \item \textbf{前端：}axios 响应拦截器统一处理业务错误码；token 失效（40101）时自动清理登录态并跳转登录页；其余错误通过 Toast 统一提示。
  \item \textbf{实时通道：}Socket 连接层使用 token 鉴权；在权限不足或参数错误时通过 \texttt{doc:error}/\texttt{board:error} 等事件回传，避免静默失败。
\end{enumerate}

\textbf{已知问题与缺陷：}
\begin{enumerate}[label=(\arabic*)]
  \item \textbf{白板接口类型校验顺序存在风险：}\texttt{board.service.js} 的部分写操作在校验权限后才判断资源类型，若误传文档 \texttt{item\_id} 且当前用户有权限，理论上可能先修改或删除数据，再返回“白板不存在”。后续应将类型校验前置到权限断言阶段。
  \item \textbf{refresh token 刷新流程未二次校验用户状态：}\texttt{auth.service.js} 中的 \texttt{refresh()} 仅校验 token 合法性并签发新的 accessToken，未再次确认用户是否已被封禁或删除，后续应补充用户状态检查。
  \item \textbf{个别源文件仍有中文编码异常：}部分注释和字符串存在乱码，不影响本次测试执行，但会降低日志可读性，建议统一转为 UTF-8 编码后再整理提交。
\end{enumerate}

\subsection{日志与监控方案}

\begin{enumerate}[label=(\arabic*)]
  \item \textbf{请求日志：}后端使用 morgan 记录请求方法、路径、状态码与耗时，为性能分析与问题回溯提供依据。
  \item \textbf{错误日志：}全局错误处理中间件在开发环境输出堆栈，便于快速定位；生产环境应接入结构化日志（如 Winston/Pino）并输出到文件或日志平台。
  \item \textbf{运行监控（建议）：}可进一步加入进程级健康检查、数据库连接监控、Socket 在线连接数统计，以及 Prometheus + Grafana 等指标体系，便于上线后观察稳定性与容量。
\end{enumerate}
